🔴 Vulnerabilidades CRÍTICAS

C-01 · Archivo .env.production commiteado en git

🔴 CRÍTICO

El archivo .env.production con las credenciales reales de Supabase fue commiteado y existe en el historial de git. Aunque el archivo se borre hoy, las credenciales quedan permanentemente en el historial y cualquiera con acceso al repo puede recuperarlas.

Evidencia — historial de git: 
commit 1eb126df1eee049e63aeea3aa27ec91ad3f41147
Author: Daniel Alberto Vázquez Mendoza
Date:   Thu Apr 30 15:17:23 2026 -0600

+VITE_SUPABASE_URL=https://lfmbhdtrxmtxjwyfzdcz.supabase.co
+VITE_SUPABASE_ANON_KEY=sb_publishable_Ne1rajOeqAJETBoXcIvA6A_l0-NBH1f
Estado actual del .gitignore: 
# .gitignore actual — solo ignora .env, NO .env.production
.env        ← ✅ ignorado
# .env.production ← ❌ FALTA — por eso entró en git
Impacto: Con la URL y ANON key de Supabase, un atacante puede hacer queries directas a tu base de datos, leer todas las propiedades, usuarios y datos del CRM, y dependiendo del RLS configurado en Supabase, manipular o borrar datos. Si alguna vez se subió la service_role key (que bypasea el RLS), el impacto sería catastrófico.
Corrección paso a paso: 
# Paso 1 — Sacar .env.production del tracking de git
git rm --cached .env.production

# Paso 2 — Actualizar .gitignore
echo ".env.production" >> .gitignore
echo ".env.local" >> .gitignore
echo ".env.*.local" >> .gitignore

# Paso 3 — Commitear
git add .gitignore
git commit -m "security: remove .env.production from git tracking"

# Paso 4 — Limpiar el historial (OBLIGATORIO)
# Instalar git-filter-repo si no lo tienes:
pip install git-filter-repo

# Eliminar el archivo de TODO el historial:
git filter-repo --path .env.production --invert-paths --force

# Paso 5 — Hacer force push (coordinar con el equipo)
git push origin --force --all

# Paso 6 — Rotar las credenciales en Supabase dashboard
# → Settings → API → Regenerar las API Keys
# Las antiguas deben considerarse comprometidas.

C-02 · Registro abierto: cualquier email puede crear una cuenta de asesor

🔴 CRÍTICO

El componente Login.jsx permite que cualquier persona en internet cree una cuenta nueva sin ningún mecanismo de validación. Además, App.jsx auto-inserta en la tabla users a cualquier usuario que se autentique con Google o magic link aunque no esté registrado previamente.

Código vulnerable — Login.jsx, handleSignUp: 
// Cualquier persona puede llegar a esta URL y crear una cuenta
const { data, error } = await supabase.auth.signUp({
  email: email.trim().toLowerCase(),
  password,          // sin validación de fortaleza
  // sin CAPTCHA, sin verificación de dominio, sin aprobación manual
});
Código vulnerable — App.jsx, checkAllowlist: 
// Si el email NO existe en la tabla users → lo CREA automáticamente
// Esto significa que cualquier email de Google entra al CRM
if (error) {
  const { data: newUser, error: insertError } = await supabase
    .from('users')
    .insert({
      email: session.user.email.toLowerCase(),
      plan: 'starter',
      active: true,      // ← activo por defecto
    })
    .select().single();
}
Impacto: Un atacante puede registrar cientos de cuentas, explorar el CRM completo con plan "starter", y potencialmente escalar privilegios si encuentra otros fallos de autorización. El texto "Exclusivo para asesores autorizados" es decorativo — no hay ningún control real.
Corrección — eliminar el auto-registro y añadir invite-only: 
// OPCIÓN A: Quitar el botón/función de registro completamente
// Solo Magic Link o Google para emails ya existentes en la tabla users.

// OPCIÓN B: Invite-only — el admin crea el usuario, luego el asesor activa
// En App.jsx, cambiar el comportamiento de auto-insert:

const checkAllowlist = async (session) => {
  const { data, error } = await supabase
    .from('users')
    .select('id, active, plan')
    .eq('email', session.user.email.toLowerCase())
    .single();

  if (error || !data) {
    // NO crear automáticamente — mostrar pantalla de "acceso pendiente"
    await supabase.auth.signOut();
    setSession(null);
    setAccessDenied(true);  // mensaje: "Tu cuenta aún no está activada"
    setAuthLoading(false);
    return;
  }

  if (!data.active) {
    await supabase.auth.signOut();
    setSession(null);
    setAccessDenied(true);
    setAuthLoading(false);
    return;
  }

  setSession(session);
  setAuthLoading(false);
};
🟠 Vulnerabilidades ALTAS

A-01 · Permisos de archivos .env demasiado abiertos

🟠 ALTO

Los archivos .env y .env.production tienen permisos 644 (lectura para todos los usuarios del sistema). En un servidor compartido o si alguien obtiene acceso al sistema de archivos, puede leer las credenciales directamente. 

-rw-r--r--@ .env             # 644 — cualquier usuario del sistema puede leer
-rw-r--r--@ .env.production  # 644 — igual de expuesto
chmod 600 .env
chmod 600 .env.production
# -rw------- → solo el propietario puede leer y escribir

A-02 · Falta de validación de fortaleza de contraseña en registro

🟠 ALTO

El formulario de registro acepta cualquier contraseña sin validar longitud, complejidad ni si está en listas de contraseñas comprometidas. El único requisito visible es "Mínimo 6 caracteres" que viene de Supabase por defecto — una contraseña como 123456 es válida. 

// Login.jsx — no hay validación de password antes de signUp
const { data, error } = await supabase.auth.signUp({
  email: email.trim().toLowerCase(),
  password,  // acepta "123456", "aaaaaa", etc.
});
// Añadir validación antes de llamar a Supabase
const validatePassword = (pwd) => {
  if (pwd.length < 8) return 'Mínimo 8 caracteres';
  if (!/[A-Z]/.test(pwd)) return 'Debe incluir al menos una mayúscula';
  if (!/[0-9]/.test(pwd)) return 'Debe incluir al menos un número';
  return null;
};

const handleSignUp = async (e) => {
  e.preventDefault();
  const pwdError = validatePassword(password);
  if (pwdError) { setErrorMsg(pwdError); return; }
  // ... resto del flujo
};
🟡 Vulnerabilidades MEDIAS

M-01 · Errores de Supabase expuestos al usuario via alert()

🟡 MEDIO

En UserManager.jsx se usa alert() con mensajes que podrían incluir información técnica interna. Aunque actualmente los mensajes son genéricos, el patrón es peligroso — un error de Supabase puede contener el nombre de tablas, columnas o estructura interna. 

// UserManager.jsx — alert directo con mensajes de sistema
alert(editingUser ? '¡Asesor actualizado!' : '¡Asesor creado exitosamente!');

// Los catch blocks silencian errores de Supabase — pueden enmascarar fallos
try {
  const { data, error } = await supabase.from('users').update(form)...
  if (error) throw error;
  saved = data;
} catch { saved = { ...form, id: editingUser.id }; }  // ← catch vacío
// Reemplazar alert() por notificaciones inline
// Nunca pasar err.message directamente al UI sin sanitizar

const handleError = (err, fallbackMsg) => {
  console.error('[UserManager]', err); // solo en dev
  setErrorMsg(fallbackMsg); // mensaje genérico al usuario
};

// Catch con logging apropiado
} catch (err) {
  handleError(err, 'No se pudo guardar. Intenta de nuevo.');
  saved = { ...form, id: editingUser.id };
}

M-02 · No hay verificación de RLS en Supabase documentada

🟡 MEDIO

El proyecto usa la ANON key de Supabase para todas las operaciones del frontend. La seguridad real de los datos depende 100% de las políticas de Row Level Security (RLS) en Supabase, pero no hay evidencia en el código o esquema de que RLS esté habilitado y correctamente configurado para todas las tablas.

Impacto: Si RLS no está habilitado en la tabla users o properties, cualquier usuario autenticado con la anon key puede leer y modificar todos los registros de todos los asesores. 
-- En Supabase SQL Editor, verificar y habilitar RLS:

-- 1. Habilitar RLS en todas las tablas
ALTER TABLE users ENABLE ROW LEVEL SECURITY;
ALTER TABLE properties ENABLE ROW LEVEL SECURITY;

-- 2. Política: cada asesor solo ve/edita sus propios datos
CREATE POLICY "asesor_read_own"
  ON users FOR SELECT
  USING (auth.uid() = id OR email = auth.jwt()->>'email');

-- 3. Solo admins pueden leer todos los usuarios
CREATE POLICY "admin_read_all"
  ON users FOR SELECT
  USING (auth.jwt()->>'role' = 'admin');

-- 4. Propiedades: cualquiera puede leer las activas (portal público)
CREATE POLICY "public_read_active_properties"
  ON properties FOR SELECT
  USING (active = true);

-- 5. Solo el asesor propietario puede editar sus propiedades
CREATE POLICY "asesor_edit_own_properties"
  ON properties FOR ALL
  USING (auth.uid() = user_id);

M-03 · console.warn expone mensajes de error de Supabase en producción

🟡 MEDIO

Hay múltiples console.warn y console.error que imprimen mensajes de error de Supabase en la consola del navegador. En producción, esto puede revelar estructura de tablas, nombres de columnas y detalles de la base de datos a cualquiera que abra las DevTools. 

// App.jsx
console.warn('Allowlist error:', err.message);         // expone error de Supabase
console.warn('Portal: usando sample data', err.message);

// UserManager.jsx
console.warn('Supabase users no disponible, usando sample data:', err.message);
console.warn('Toggle user persist failed:', err.message);
// Usar una función que solo loguea en desarrollo
const devLog = (...args) => {
  if (import.meta.env.DEV) console.warn(...args);
};

// Reemplazar todos los console.warn en producción
devLog('Allowlist error:', err.message);
🔵 Vulnerabilidades BAJAS

B-01 · URL de avatar sin validación (posible Open Redirect / SSRF)

🔵 BAJO

El campo avatar_url acepta cualquier URL sin validar que pertenezca a un dominio conocido. Un admin malintencionado podría insertar una URL que apunte a recursos internos o a contenido inapropiado que se renderizaría en el CRM. 

// Validar que avatar_url sea una URL HTTPS de dominio conocido
const ALLOWED_AVATAR_DOMAINS = ['images.unsplash.com', 'lh3.googleusercontent.com', 'storage.googleapis.com'];

const validateAvatarUrl = (url) => {
  if (!url) return true; // campo opcional
  try {
    const parsed = new URL(url);
    return parsed.protocol === 'https:' && ALLOWED_AVATAR_DOMAINS.some(d => parsed.hostname.endsWith(d));
  } catch {
    return false;
  }
};

B-02 · Sin Content Security Policy (CSP) headers

🔵 BAJO

El proyecto no configura headers de seguridad HTTP. En producción (Vercel), estos se configuran en vercel.json. Sin CSP, si existiera un XSS sería más fácil de explotar. 

// vercel.json — añadir headers de seguridad
{
  "headers": [
    {
      "source": "/(.*)",
      "headers": [
        { "key": "X-Frame-Options", "value": "DENY" },
        { "key": "X-Content-Type-Options", "value": "nosniff" },
        { "key": "Referrer-Policy", "value": "strict-origin-when-cross-origin" },
        { "key": "Permissions-Policy", "value": "camera=(), microphone=(), geolocation=()" },
        {
          "key": "Content-Security-Policy",
          "value": "default-src 'self'; script-src 'self' 'unsafe-inline'; style-src 'self' 'unsafe-inline' https://fonts.googleapis.com; img-src 'self' data: https: blob:; font-src 'self' https://fonts.gstatic.com; connect-src 'self' https://*.supabase.co;"
        },
        { "key": "Strict-Transport-Security", "value": "max-age=63072000; includeSubDomains; preload" }
      ]
    }
  ]
}
⚪ Observaciones INFO

I-01 · La ANON key de Supabase en VITE_* es visible en el bundle del frontend

⚪ INFO

Las variables VITE_* se embeben en el bundle JavaScript que se sirve al navegador — esto es por diseño en Vite. La VITE_SUPABASE_ANON_KEY (sb_publishable_) es la clave pública de Supabase, diseñada para ser expuesta. No es un fallo siempre que RLS esté correctamente configurado (ver M-02). Nunca usar una service_role key como variable VITE_.

I-02 · No hay .env.example en el repositorio

⚪ INFO

No existe un archivo .env.example con las variables necesarias sin valores reales. Esto dificulta el onboarding de colaboradores y puede llevar a que alguien copie el .env.production real por no saber qué variables se necesitan. 

# Crear .env.example con valores placeholder
VITE_SUPABASE_URL=https://tu-proyecto.supabase.co
VITE_SUPABASE_ANON_KEY=tu-anon-key-aqui

I-03 · Sin rate limiting en el frontend para los formularios de auth

⚪ INFO

No hay protección en el cliente contra spam de intentos de login o envío masivo de magic links. Supabase tiene rate limiting propio por IP, pero es configurable y podría no ser suficiente. Considerar añadir un honeypot field o CAPTCHA si el abuso se convierte en problema.
